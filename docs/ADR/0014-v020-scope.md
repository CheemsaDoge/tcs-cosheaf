# ADR 0014: v0.2.0 Scope

## Status

Proposed

## Context

After the `v0.1.1` formal-link metadata baseline, current `main` contains
additional release-hardening work across retrieval, context packs, local
orchestrator dry-runs, optional verifier adapters, evaluation, observability,
source ingestion, and developer tooling boundaries.

The project needs a next milestone that is useful for a demo-ready MVP without
turning the scaffold into a production hosted agent platform or overstating
formal-verification capability. The risky alternative is to let `v0.2.0`
absorb every promising direction: hosted LLM workers, web UI, automatic
theorem proving, automatic accepted promotion, and broad Lean/mathlib/CSLib
integration. That would weaken the existing review, gate, promotion, and
public/private KB boundaries.

## Decision

Propose `v0.2.0` as a bounded local-MVP milestone. It should package, polish,
and document the deterministic workflow already represented on `main`.

Included scope:

1. Librarian v1 usability and evaluation.
   - Deterministic artifact-card retrieval.
   - Public/private filtering.
   - Retrieval audit records.
   - Clear issue-scoped context assembly docs and CLI ergonomics.
2. Context pack v2.
   - Card-first default context handoff.
   - Explicit full-artifact pull budgets.
   - Policy-safe public-only behavior.
   - Conservative formal-link display that does not claim proof.
3. Local orchestrator state machine.
   - Replayable issue-scoped plans.
   - Task DAGs, reducers, worker-bundle validation, and local dry runs.
   - No hosted LLM default execution.
4. Fake provider model interface.
   - Deterministic provider-neutral tests.
   - Explicit capability negotiation before any hosted provider adapter.
5. Retrieval evaluation harness.
   - Regression metrics for relevance, forbidden hits, private leakage, and
     accepted-priority behavior.
6. Optional external Lean `#check` ergonomics if stable.
   - Lean/lake must remain optional.
   - Missing tools remain `skipped`, not `pass`.
   - A successful `#check` means only that the import and symbol resolved.

## Non-Goals

`v0.2.0` must not include:

- Production hosted multi-agent runtime.
- Hosted LLM execution enabled by default.
- Automatic theorem proving.
- Automatic accepted-artifact promotion.
- Web UI.
- Multi-user authentication or permissions.
- Full CSLib/mathlib ingestion or vendoring.
- Automatic informal/formal semantic-alignment checking.
- Any change that lets worker output, AI review, tool output, retrieval scores,
  generated dashboards, or sidecars become accepted knowledge.

## Consequences

The milestone remains demo-ready and reviewable. It gives downstream users a
coherent local workflow: mount public KB, keep private work in a writable
overlay, validate, gate, index, retrieve, build context, and run local dry-run
worker workflows without confusing those workflows with accepted knowledge or
machine-checked mathematical proof.

The milestone keeps hosted provider work, web UI, multi-user auth, broader
formal-library integration, and production agent-runtime concerns as future
separate proposals. Each future proposal must preserve review, gate,
promotion, public/private KB, and formal-alignment boundaries.

## Follow-Up Requirements

Before cutting a `v0.2.0` tag, run a release audit across the framework,
public KB, and workspace template:

- Check README, roadmap, release checklist, current milestone, and project
  state docs for stale claims.
- Run the full framework verification ladder.
- Run workspace-template demo and public-KB validation/gates against the
  intended framework tag.
- Confirm formal-link language still distinguishes metadata, `#check`, and
  human alignment review.
- Confirm no accepted public artifact lacks source metadata and human review.
