# 审查策略

## 目的

TCS-Cosheaf 将 Git 视为持久的项目记忆。审查策略通过确保代码库变更限定在问题（issue）范围内，并经过验证、审查，最后通过可审计的流程合并，来保护这些记忆。

此策略适用于所有变更，包括人工编写的变更和 Codex 辅助的变更。

## 保护主分支

`main` 分支必须在 GitHub 中受到保护。

不允许直接推送到 `main`。变更只能通过满足以下必填检查和审查要求的拉取请求（pull requests）进入 `main`。

`main` 的分支保护或代码库规则集应要求：

- 合并前需提交拉取请求；
- 合并前需通过必须的状态检查；
- 合并前分支必须是最新的；
- 基础状态检查名称为 `lint`、`typecheck`、`test`、`validate` 和 `gate`。

## 必需的变更流程

所有变更必须遵循以下顺序：

1. 打开或选择一个说明目标、范围、允许的文件、验收标准和所需命令的 issue。
2. 从当前的 `main` 分支创建一个任务分支。
3. 仅在该分支上实现该 issue 的内容。
4. 打开一个目标为 `main` 的拉取请求。
5. 运行 CI 和关卡（gatekeeper）检查。
6. 审查该拉取请求。
7. 只有在必需的检查通过且满足审查要求后才能合并。

除非 issue 明确拆分为更小的后续 issue，否则一个 issue 通常对应一个分支和一个拉取请求。

## 基础检查

基础的本地和 CI 检查包括：

- `make lint`
- `make typecheck`
- `make test`
- `make validate`
- `make gate`

CI 必须在合并前通过。在可行的情况下，本地运行的记录应写在拉取请求正文中。不得隐藏、忽略验证或关卡失败，也不得将其报告为成功。

可选的外部形式化工具（如 Lean、SAT 求解器、SMT 求解器或 Sage）必须保持为可选。它们的缺失应该产生跳过（skipped）的验证器结果，而不是核心验证的崩溃。

## 必须更新文档

公共接口的更改需要更新 `context/INTERFACE_REGISTRY.md`。公共接口包括 CLI 命令、Python API、制品模式（schemas）、关卡结果格式、验证器适配器契约、上下文包格式以及记录的文件布局。

架构变更需要在 `docs/ADR/` 目录下添加或更新 ADR（架构决策记录）。

工作流、命令、制品模式、状态、关卡、验证器或审查行为的变更必须在同一个拉取请求中更新相关文档。

## 需要人工审查的领域

以下领域在合并前需要人类维护者审查：

- 制品模式（schema）变更；
- 制品状态格（lattice）或状态/路径规则变更；
- 关卡检查、报告模式或退出代码行为；
- 依赖图行为；
- 代码库索引模式、清单格式或重建行为；
- 知识库的 accepted/draft/refuted/obsolete 状态的不变量；
- 验证器结果语义，特别是 `pass`、`fail`、`error` 和 `skipped`；
- 需要注册表更新的公共接口变更；
- 需要 ADR 的架构变更。

Codex 审查可以通过识别风险、缺失测试、不确定性或违反策略来提供帮助。它不取代维护者的判断。维护者仍负责决定拉取请求是否正确、是否经过了充分测试以及合并是否安全。

## 合并拦截条件

在以下任何情况为真时，不得合并拉取请求：

- 必需的 CI 状态检查失败或缺失；
- `make validate` 或 `make gate` 失败；
- 未经合理解释就隐藏或降级验证失败；
- 当要求确定性时，生成的输出却是不确定的；
- 公共接口的变更缺少对 `context/INTERFACE_REGISTRY.md` 的更新；
- 架构的变更缺少 ADR 更新；
- 模式（schema）、状态、关卡或索引的变更未经过人工审查；
- 分支已过时，而分支保护规则要求它是最新的。
