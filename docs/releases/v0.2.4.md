# v0.2.4 - Artifact Failure Memory + Attempt Traceability

Status: published release. Package metadata and `cosheaf.__version__` record
`0.2.4`, the annotated public `v0.2.4` tag is published, the GitHub release is
published, release smoke from `@v0.2.4` passed, and downstream active pins in
`tcs-cosheaf-workspace-template` and `tcs-kb-public` now use `v0.2.4`.

This release packages the Artifact Failure Memory + Attempt
Traceability work that landed after the published `v0.2.3` Verification
Evidence Hardening release. The milestone adds a structured, optional
artifact-level `failure_log` surface for failed attempts, dead directions,
blocked approaches, and lessons learned while preserving all review, verifier,
gate, promotion, provider, MCP, public/private, and source-review boundaries.

This release is not production-ready software. It does not add automatic
theorem proving, full Lean/mathlib/CSLib proof checking of informal statements,
automatic autoformalization, automatic informal/formal semantic alignment,
automatic accepted promotion, AI-as-human-review, a web UI, SaaS behavior,
multi-user permissions, default-on provider calls, or provider/MCP authority
expansion.

## Highlights

- Artifact-level failure memory:
  `BaseArtifact.failure_log` is an optional, backward-compatible list of
  typed `FailureLogEntry` records. Entries capture attempted direction,
  failure reason, origin, attempt kind, evidence paths, related verifier or
  candidate-counterexample references, next possible directions, status, and
  limitations.
- Read-only inspection:
  `cosheaf artifact failures <artifact-id> --json` returns deterministic
  failure-log payloads with artifact path, root scope, readonly metadata,
  failure entries, and an explicit non-authority notice.
- Controlled draft writes:
  `cosheaf artifact failure add --artifact <artifact-id> --input-json <path>
  --json` and `--dry-run` append only to writable draft/pre-accepted artifacts,
  refuse accepted paths/statuses and readonly roots, and reject authority
  spoofing.
- WorkerBundle bridge:
  `cosheaf artifact failure plan-from-bundle` and `add-from-bundle` convert
  WorkerBundle v2 failed attempts into proposed or controlled failure-log
  entries. Provider/agent origin is preserved, and typed counterexample
  candidates remain linked by reference only.
- Retrieval and context surfacing:
  artifact cards, memory search, and context packs surface failure memory as
  review context. Context packs render `Known Failed Directions` sections only
  for visible cards, and public-only context excludes private failure-log text.
- Promotion-readiness warnings:
  read-only promotion-readiness reports include `unresolved_failure_memory`
  warning reasons without changing promotion behavior or treating failure
  memory as verifier failure evidence.
- Workspace and public KB policy:
  workspace-template includes a failure-memory demo that writes only under
  ignored runtime copies, and public KB policy explains how reviewed failure
  memory may be used without encouraging unreviewed accepted content.
- Security and eval coverage:
  tests cover authority spoofing, accepted-path evidence rejection,
  public/private leakage, retrieval recall, repeated failed-direction
  detection, authority boundaries, and candidate-counterexample mislabel
  prevention.

## Boundaries

- `failure_log` is research memory only. It is not proof, verifier success,
  checked counterexample evidence, human review, source review, gate success,
  accepted status, accepted refutation, or promotion evidence.
- Failure memory can explain why a direction failed, but it does not refute a
  claim unless a separate checked counterexample or reviewed refutation path
  exists.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, verifier evidence where applicable, source metadata for
  accepted public artifacts, and explicit promotion.
- Skipped verifier, provider, SAT, SMT, Lean, lake, optional-tool, or network
  results are not passes.
- A successful Lean `#check` means only import and symbol resolution; it does
  not prove informal/formal semantic alignment.
- Provider real-run behavior remains explicit and default-off.
- MCP remains optional adapter work and is not the primary release path.
- Public KB accepted content still requires source metadata and human review.

## Downstream Guidance

Downstream workspaces that need artifact failure-memory support can pin to:

```text
git+https://github.com/CheemsaDoge/tcs-cosheaf.git@v0.2.4
```

Use `tcs-cosheaf-workspace-template` as the user-facing entry point. Mount or
checkout `tcs-kb-public` as readonly public knowledge and keep private work in
the writable private KB overlay. Do not manually merge framework, public KB,
and private workspace repositories into one mixed tree.

## Readiness Audit Summary

The readiness audit found no release blocker for entering this release
candidate. It confirmed:

- artifact-level `failure_log` is implemented and backward compatible;
- existing artifacts still validate;
- failure entries are documented and surfaced as non-authoritative;
- context packs surface failed directions without leaking private data;
- promotion readiness mentions unresolved failures without changing promotion
  rules;
- WorkerBundle and review-request bridges preserve review-only boundaries;
- workspace and public KB policy surfaces are updated; and
- all three repositories had no open issues or PRs before the audit issue was
  created.

Deferred non-blockers:

- Checked-counterexample review artifacts beyond WorkerBundle candidate records
  remain future evidence-taxonomy work.
- Optional SAT, SMT, Lean, lake, hosted-provider, and network paths remain
  optional; unavailable tools or network rows must stay `skipped`, not `pass`.

## Known Limitations

- No hosted provider call is default-on.
- No real provider call runs in CI or default tests.
- SAT, SMT, plain Lean, and external Lean reference checking remain minimal
  optional invocation paths.
- External Lean library reference checking does not fetch CSLib/mathlib or
  prove semantic alignment.
- Checked-counterexample review artifacts beyond WorkerBundle candidate
  records remain future work.
- Failure memory is useful review context, not accepted mathematical truth.
- External public KB integration remains through local workspace roots, not a
  hosted registry service.
