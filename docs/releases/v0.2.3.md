# v0.2.3 - Verification Evidence Hardening

Status: release candidate. Package metadata and `cosheaf.__version__` record
`0.2.3` on the release-candidate branch. The public `v0.2.3` tag, GitHub
release, and downstream repository pin updates are intentionally deferred until
after the release-candidate PR merges and the maintainer release action verifies
the tag target and release smoke.

`v0.2.3` packages the verification-evidence hardening work that landed after
the published `v0.2.2` Provider Transport + Agent Workflow Hardening release.
The milestone strengthens verifier evidence records, optional SAT/SMT/Lean
result-depth coverage, counterexample candidate handling, read-only promotion
readiness, failure-preserving review requests, verifier-evidence evals, and
three-repository readiness checks.

This release candidate is not production-ready software. It does not add
automatic theorem proving, full Lean/mathlib/CSLib proof checking of informal
statements, automatic autoformalization, automatic informal/formal semantic
alignment, automatic accepted promotion, AI-as-human-review, a web UI, SaaS
behavior, multi-user permissions, default-on provider calls, or provider/MCP
authority expansion.

## Highlights

- Verifier evidence record v1:
  `VerifierEvidenceRecord` serializes verifier outputs with stable evidence
  IDs, verifier kind, tool metadata, command/cwd metadata, normalized
  `pass`/`fail`/`error`/`skipped` result state, reason code, log paths,
  optional checker hashes, timestamps, and limitations. It is evidence
  serialization support only; it is not human review, proof of mathematical
  truth, or promotion authority.
- Read-only promotion readiness:
  `cosheaf promotion readiness --artifact <artifact-id> --json` and
  `cosheaf promotion readiness --issue <issue-id> --json` explain missing
  review, failed or skipped verifier evidence, source metadata gaps,
  dependency risk, private dependencies, draft status, readonly roots, and
  repository gate blockers without promoting anything.
- Optional SAT/SMT/Lean result-depth coverage:
  SAT, SMT, plain Lean, and external Lean library reference paths have
  fake-backend, mocked, local-fixture, or tool-absence coverage. CI and default
  tests do not require `kissat`, `z3`, `lean`, `lake`, CSLib, mathlib, network
  access, or API keys.
- Lean external reference ergonomics:
  the external Lean reference path preserves clearer manifest diagnostics and
  missing import/symbol stderr logs in fake-backend coverage. A successful
  generated `import <module>` / `#check <symbol>` run means only that the
  configured Lean environment resolved the import and symbol.
- Typed counterexample candidates:
  WorkerBundle v2 can carry review-only `counterexample_candidates` with
  candidate IDs, target claims, construction summaries, evidence paths,
  verifier-request IDs, statuses, and limitations. Candidates do not become
  accepted refutations by themselves.
- Failure-preserving review requests:
  `cosheaf review request-from-bundle --bundle <path>` writes or previews draft
  informational review-request records from WorkerBundle v2 assumptions,
  uncertainty, failed attempts, verifier requests, counterexample candidates,
  risk flags, next steps, and limitations.
- Verifier evidence eval:
  the deterministic eval suite covers pass evidence supporting readiness only
  when policy allows, failed evidence blocking readiness, skipped required
  evidence remaining not-a-pass, candidate counterexamples staying review-only,
  and Lean `#check` staying symbol/import resolution only.
- Three-repository readiness matrix:
  `scripts/ecosystem_smoke.py --matrix` covers framework verifier-evidence eval
  smoke, optional verifier availability, workspace-template verifier-evidence
  demo coverage, public KB policy guard, and public KB verifier-policy
  self-test coverage. Network rows remain opt-in.

## Boundaries

- A `verifier_request` is not a `verifier_result`.
- A `candidate_counterexample` is not a checked or accepted refutation.
- A skipped verifier, provider, SAT, SMT, Lean, lake, or network row is not a
  pass.
- A successful Lean `#check` means only import and symbol resolution in the
  configured Lean environment.
- Lean `#check` does not prove informal/formal semantic alignment.
- Formal links remain metadata unless a checker records a result.
- Validation, gates, evals, verifier output, provider output, generated review
  requests, and AI review are not human review.
- Promotion-readiness reports are advisory and read-only. They do not write
  accepted artifacts, set `status: accepted`, create human review, or replace
  `cosheaf artifact promote`.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, verifier evidence where applicable, source metadata for
  accepted public artifacts, and explicit promotion.
- Public KB accepted content still requires source metadata and human review.
- Provider real-run behavior remains explicit and default-off.
- MCP remains optional adapter work and is not the primary release path.

## Downstream Guidance

After the public `v0.2.3` tag is published and release smoke passes, downstream
workspaces that need verification-evidence hardening should pin to:

```text
git+https://github.com/CheemsaDoge/tcs-cosheaf.git@v0.2.3
```

Downstream repositories that only need the earlier Formal Link Layer support
can remain on `v0.1.1`. Repositories that need the deterministic local-MVP
workflow can remain on `v0.2.0`. Repositories that need CLI-agent and
hosted-provider gateway prerelease surfaces can remain on `v0.2.1`.
Repositories that need provider transport and agent workflow hardening can
remain on `v0.2.2`.

Use `tcs-cosheaf-workspace-template` as the user-facing entry point. Mount or
checkout `tcs-kb-public` as readonly public knowledge and keep private work in
the writable private KB overlay. Do not manually merge framework, public KB,
and private workspace repositories into one mixed tree.

## Release-Candidate Verification

The release-candidate PR must run and report:

```bash
python -m cosheaf.cli version --json
make lint
make typecheck
make test
make validate
make gate
python scripts/ecosystem_smoke.py --matrix --cosheaf "python -m cosheaf.cli" --framework-root . --workspace-template-root ../tcs-cosheaf-workspace-template --public-kb-root ../tcs-kb-public --json
git diff --check
```

The default no-network ecosystem matrix should report the framework git-tag
release smoke and workspace-template install demo as `skipped` unless the
operator supplies `--include-network`. The optional verifier availability row
may also be `skipped` when SAT/SMT/Lean/lake tools are unavailable. Skipped
rows must remain separate from passes.

Before a public release announcement, an operator should also run the
network-enabled matrix rows with `--include-network` when package install and
GitHub clone access are available. Those rows remain separate from default CI
and must not be described as passed unless they actually run and pass.

## Publication Checklist

Publication is a maintainer release action after this release-candidate PR
merges:

```bash
git fetch origin main --tags
git checkout main
git pull --ff-only
git tag -a v0.2.3 -m "v0.2.3 Verification Evidence Hardening"
git push origin v0.2.3
python scripts/release_smoke.py --source git+https://github.com/CheemsaDoge/tcs-cosheaf.git@v0.2.3
```

If a `v0.2.3` tag already exists but points to an unexpected commit, stop and
do not force-retag without explicit maintainer approval. Downstream pin updates
belong after tag publication and release smoke succeed.

## Known Limitations

- The `v0.2.3` tag and GitHub release are not created by this PR.
- Downstream repositories are not updated by this PR.
- No hosted provider call is default-on.
- No real provider call runs in CI or default tests.
- SAT, SMT, plain Lean, and external Lean reference checking remain minimal
  optional invocation paths.
- External Lean library reference checking does not fetch CSLib/mathlib or
  prove semantic alignment.
- Checked-counterexample review artifacts beyond WorkerBundle candidate
  records remain future work.
- External public KB integration remains through local workspace roots, not a
  hosted registry service.
