# v0.2.3 - Verification Evidence Hardening

Status: release readiness audit draft. This is not a published release note
yet. Package metadata and `cosheaf.__version__` still report `0.2.2`; the
`v0.2.3` version bump, tag, GitHub release, and downstream pin updates belong
to a later release-candidate task after this audit is merged and re-verified.

`v0.2.3` packages the verification-evidence hardening work that landed after
the published `v0.2.2` Provider Transport + Agent Workflow Hardening release.
The milestone strengthens verifier evidence records, optional SAT/SMT/Lean
result-depth coverage, counterexample candidate handling, read-only promotion
readiness, failure-preserving review requests, verifier-evidence evals, and
three-repository readiness checks.

This release line is not production-ready software. It does not add automatic
theorem proving, full Lean/mathlib/CSLib proof checking of informal
statements, automatic autoformalization, automatic informal/formal semantic
alignment, automatic accepted promotion, AI-as-human-review, a web UI, SaaS
behavior, multi-user permissions, default-on provider calls, or provider/MCP
authority expansion.

## Readiness Audit Result

The current main branch can enter the `v0.2.3` release-candidate task after
this audit PR merges and the required verification ladder passes, provided no
new blocking issue or PR appears before the release-candidate branch starts.

The release-candidate task should still do the actual release preparation:
update package metadata from `0.2.2` to `0.2.3`, convert this draft into final
release notes, run the required command ladder, run the ecosystem readiness
matrix, and only then create the tag/release and downstream pin updates.

## Audit Questions

1. Are verifier evidence records stable?

   Yes for the current scope. `VerifierEvidenceRecord` v1 serializes verifier
   outputs with stable evidence IDs, verifier kind, tool metadata, command/cwd
   metadata, normalized `pass`/`fail`/`error`/`skipped` result state, reason
   code, log paths, optional checker hashes, timestamps, and limitations. It is
   evidence serialization support only; it is not human review, promotion
   authority, or proof of mathematical truth.

2. Are SAT/SMT/Lean optional paths tested without mandatory tools?

   Yes. SAT, SMT, plain Lean, and external Lean library reference coverage uses
   fake backends, mocked command runners, local fixtures, or tool-absence
   skips. CI and default tests do not require `kissat`, `z3`, `lean`, `lake`,
   CSLib, mathlib, network access, or API keys.

3. Does skipped-not-pass remain enforced?

   Yes. Verifier adapters distinguish `skipped` from `pass`; promotion
   readiness reports skipped required evidence as a blocker; evals count
   skipped-not-pass boundaries; and the ecosystem matrix counts optional
   verifier unavailability and default network rows as `skipped`, not pass.
   The optional verifier availability probe uses exit code `77` for skipped
   matrix rows.

4. Are counterexamples still candidates unless checked and reviewed?

   Yes. WorkerBundle v2 `counterexample_candidates` remain review-only records.
   Proposed and needs-check candidates are not checked refutations. Even
   checked candidate statuses require evidence paths and still do not write
   accepted knowledge, create verifier results, mark human review, or promote
   artifacts by themselves.

5. Are promotion-readiness reports read-only?

   Yes. `cosheaf promotion readiness` reports
   `accepted_write_performed: false`, writes only ordinary runtime gate reports
   under `.cosheaf/reports/`, and does not move artifacts, set accepted status,
   create human review, satisfy source review, or replace
   `cosheaf artifact promote`.

6. Are workspace/public KB compatibility paths clean?

   Yes for local no-network readiness. The ecosystem matrix covers the
   framework checkout, workspace-template demos, public KB policy guard, public
   KB verifier-policy self-test, readonly public KB roots, writable private KB
   overlays, private-to-public dependency allowance, public-to-private
   dependency rejection, accepted-to-draft dependency rejection, validation,
   gatekeeper, index rebuild, context build, and verifier-evidence demo
   coverage. Default matrix rows do not clone remote repositories or perform
   network installs.

7. Are open issues/PRs empty or intentionally deferred?

   At the start of this audit, the only open issue was the audit issue itself
   and there were no open PRs. The known intentionally deferred item is a
   dedicated checked-counterexample review artifact or report path beyond
   WorkerBundle candidate records. Network-enabled matrix rows remain an
   operator/release-announcement check through `--include-network`, not a
   default CI requirement.

## Boundaries

- A `verifier_request` is not a `verifier_result`.
- A `candidate_counterexample` is not a checked or accepted refutation.
- A skipped verifier, provider, SAT, SMT, Lean, lake, or network row is not a
  pass.
- A successful Lean `#check` means only import and symbol resolution in the
  configured Lean environment.
- Lean `#check` does not prove informal/formal semantic alignment.
- Formal links remain metadata unless a checker records a result.
- Validation, gates, evals, verifier output, provider output, generated review
  requests, and AI review are not human review.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, verifier evidence where applicable, source metadata for
  accepted public artifacts, and explicit promotion.
- Public KB accepted content still requires source metadata and human review.

## Release-Candidate Verification

The release-candidate PR should run and report:

```bash
make lint
make typecheck
make test
make validate
make gate
python scripts/ecosystem_smoke.py --matrix --cosheaf "python -m cosheaf.cli" --framework-root . --workspace-template-root ../tcs-cosheaf-workspace-template --public-kb-root ../tcs-kb-public --json
git diff --check
```

Before a public release announcement, an operator should also run the
network-enabled matrix rows with `--include-network` when package install and
GitHub clone access are available. Those rows remain separate from default CI
and must not be described as passed unless they actually run and pass.

## Known Limitations

- The current package version is still `0.2.2` until the release-candidate task
  updates package metadata.
- No hosted provider call is default-on.
- No real provider call runs in CI or default tests.
- SAT, SMT, plain Lean, and external Lean reference checking remain minimal
  optional invocation paths.
- External Lean library reference checking does not fetch CSLib/mathlib or
  prove semantic alignment.
- Checked-counterexample review artifacts beyond WorkerBundle candidate
  records remain future work.
- External public KB integration remains through local workspace roots, not a
  hosted registry service.
