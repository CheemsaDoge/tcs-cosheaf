# v0.2.2 - Provider Transport + Agent Workflow Hardening Release Candidate

Status: release candidate. The package metadata records `0.2.2`; the public
Git tag should be created only after the release-candidate PR and required
checks pass.

`v0.2.2` packages the post-`v0.2.1` provider and workflow hardening work as a
conservative framework release candidate. It adds a controlled real-provider
opt-in path and strengthens provider privacy, failure preservation, eval, and
three-repository smoke coverage without changing the accepted-knowledge
governance boundary.

This release candidate is not a production hosted multi-agent platform. It does
not add default real provider calls, live provider calls in CI, hosted worker
CLI commands, provider MCP tools, controlled-write MCP, automatic theorem
proving, automatic Lean/mathlib/CSLib semantic alignment, automatic accepted
promotion, AI-as-human-review, a web UI, SaaS behavior, or multi-user
permissions.

## Highlights

- Optional OpenAI-compatible HTTP transport:
  a stdlib transport object exists for explicitly configured and injected
  OpenAI-compatible calls. It is default-off, requires environment-provided
  credentials, and is tested through fake, mocked, or local non-live-network
  fixtures.
- Explicit provider `real-run` CLI:
  `cosheaf provider real-run --input-json <path> --provider openai-compatible
  --confirm-send --allow-network --json` fails closed without inline context
  preview, send confirmation, network permission, endpoint/API-key
  configuration, and required private-context consent.
- Context-send policy matrix:
  provider previews stay metadata-only and cards-only under the implemented
  boundary. Public mode exposes public scope only. Private scope requires
  `private_research`, `public_only=false`, and explicit private-context
  consent.
- Provider log redaction and leak scanning:
  generated provider run records are redacted, and deterministic scanner tests
  cover API-key-shaped values, bearer tokens, environment dumps, secret-looking
  key/value pairs, hidden reasoning markers, unapproved private context, and
  avoidable private paths.
- WorkerBundle failure/counterexample hardening:
  WorkerBundle v2 preserves assumptions, uncertainty, failed attempts,
  candidate counterexamples, verification requests, dependency questions, risk
  flags, and next steps as review-only material.
- Role contract hardening:
  provider-facing role contracts require uncertainty and failure structure,
  separate verifier concerns from tool results, distinguish candidate from
  verified counterexamples, separate formal symbol resolution from semantic
  alignment, and forbid librarian claim invention.
- Provider malformed-output recovery:
  malformed JSON and schema-invalid WorkerBundle output remain
  `provider_output_validation_failed`; an optional deterministic retry can add
  a schema reminder without coercing malformed output into knowledge.
- Deterministic evals:
  provider-workflow and failure/counterexample eval suites expose metrics for
  policy denials, validation rejection, secret leaks, malformed-output
  rejection, bundle validity, failure preservation, uncertainty fields,
  counterexample candidate flags, verifier requests, and accepted-write
  violations.
- Ecosystem smoke matrix:
  `scripts/ecosystem_smoke.py --matrix` reports structured framework,
  workspace-template, and public-KB compatibility rows, with network rows
  skipped by default and not counted as passes.
- Downstream docs:
  workspace-template provider setup docs and public-only preview smoke explain
  fake/mocked/real provider boundaries without adding real provider calls to
  default demos. Public KB source-note/backlog docs were refreshed, and one
  draft foundation artifact was tightened without accepted promotion.
- Optional MCP review:
  the read-only MCP stdio surface remains optional. No new MCP tools,
  controlled-write MCP, provider MCP tools, arbitrary shell, or accepted-write
  paths are added.

## Boundaries

- No default real provider calls.
- No real provider calls in CI or default tests.
- No API keys, bearer tokens, raw secrets, hidden reasoning, full environment
  dumps, or unapproved private context may be committed or logged.
- Provider output is untrusted. It may become WorkerBundle output, typed
  review-only sub-result, draft proposal, controlled draft input, or review
  context only.
- Provider, MCP, retrieval, context, validation, gate, SAT, SMT, Lean, or eval
  output is not human review.
- AI review is not human review.
- Validation and gate success are not accepted status.
- Skipped verifier/provider/tool results are not passes.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, verifier evidence where applicable, and explicit
  promotion.
- Candidate counterexamples are not accepted refutations until checked and
  reviewed.
- Verification requests are not verifier results.
- Formal links remain metadata unless a checker records a result. A successful
  Lean `#check` means import and symbol resolution only; it does not prove
  informal/formal semantic alignment.
- MCP remains optional adapter work. Controlled-write MCP is not planned unless
  a separate maintainer-approved issue explicitly reopens it.

## Downstream Guidance

After the `v0.2.2` tag is published, downstream workspaces that need provider
transport hardening, provider-preview policy checks, provider log scanning,
failure/counterexample workflow preservation, and the ecosystem smoke matrix
should pin to:

```text
git+https://github.com/CheemsaDoge/tcs-cosheaf.git@v0.2.2
```

Downstream repositories that only need the earlier Formal Link Layer support
can remain on `v0.1.1`. Repositories that need the deterministic local-MVP
workflow can remain on `v0.2.0`. Repositories that need only CLI-agent and
hosted-provider gateway prerelease surfaces can remain on `v0.2.1`.

Use `tcs-cosheaf-workspace-template` as the user-facing entry point. Mount or
checkout `tcs-kb-public` as readonly public knowledge and keep private work in
the writable private KB overlay. Do not manually merge framework, public KB,
and private workspace repositories into one mixed tree.

## Release-Candidate Verification

The release-candidate PR must run and report:

```bash
make lint
make typecheck
make test
make validate
make gate
python scripts/ecosystem_smoke.py --matrix --framework-tag v0.2.2 --cosheaf "python -m cosheaf.cli" --framework-root . --workspace-template-root ..\tcs-cosheaf-workspace-template --public-kb-root ..\tcs-kb-public --json
git diff --check
```

The default no-network ecosystem matrix should report the framework git-tag
release smoke and workspace-template install demo as `skipped` unless the
operator supplies `--include-network`. Skipped rows must remain separate from
passes.

Network-enabled release-smoke rows may be run before a public release
announcement. If the operator cannot run those rows in the release-candidate
environment, record the exact reason and do not count them as passes.
