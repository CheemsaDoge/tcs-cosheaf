# v0.2.2 - Provider Transport + Agent Workflow Hardening Readiness Audit

Status: draft readiness audit, not a release tag.

This document records whether the current `main` branch is ready to prepare a
`v0.2.2` release-candidate PR. It does not change package metadata, create a
tag, or claim production readiness.

## Verdict

`main` is ready to enter the `v0.2.2` release-candidate task after this audit
PR merges, provided the release-candidate PR reruns the full framework command
ladder and ecosystem smoke. No blocking follow-up issue is required by this
audit.

The current package version remains `0.2.1` until the release-candidate PR
updates version metadata. The `v0.2.2` release must remain conservative:

- no default real provider calls;
- no real provider calls in CI;
- no automatic accepted promotion;
- no AI review as human review;
- no production hosted multi-agent claim;
- MCP remains optional, and controlled-write MCP remains out of scope unless a
  separate maintainer-approved issue reopens it.

## Readiness Questions

| Question | Answer | Evidence |
| --- | --- | --- |
| Is optional real OpenAI-compatible HTTP transport implemented? | Yes. | `OpenAICompatibleHttpTransport` and `OpenAICompatibleProvider` are implemented in `cosheaf/agent/providers/__init__.py`; tests cover the stdlib HTTP transport and OpenAI-compatible provider paths. |
| Is it default-off? | Yes. | Provider docs state the transport is inert unless explicitly configured and injected. The real-run CLI requires explicit input, provider config, inline context preview, `--confirm-send`, and `--allow-network`. |
| Do CI/tests avoid real provider calls? | Yes. | Provider, hosted-worker, eval, and security tests use deterministic fake providers, injected mocked transports, or local non-live-network fixtures. No test requires a live provider account or API key. |
| Are provider logs redacted? | Yes. | Provider run logging applies redaction helpers and the provider-log leak scanner has synthetic-leak and redacted-log regression tests. |
| Is context preview mandatory before real send? | Yes. | `provider real-run` requires an input envelope with inline `context_preview`, and provider docs/security docs require context preview plus operator consent before send. |
| Are failure/counterexample workflows covered? | Yes. | `cosheaf.evals.failure_counterexample`, `evals/failure_counterexample/cases.yaml`, and `tests/evals/test_failure_counterexample_eval.py` cover failed attempts, uncertainty, candidate counterexamples, verifier requests, and accepted-write boundary. |
| Are workspace docs updated? | Yes. | Workspace-template PRs #46 and #48 documented real-provider setup and added a public-only provider preview smoke without adding a real provider call to default demos. |
| Is public KB unaffected except explicitly approved policy/content PRs? | Yes. | Public KB PR #55 refreshed source-note/backlog policy docs only. PR #57 updated exactly one draft artifact, `definition.walk`, and did not add or promote accepted artifacts. |
| Is MCP still optional? | Yes. | MCP optional-adapter review PR #256 kept the stdio surface read-only, added no new tools, and kept controlled-write MCP and provider MCP tools out of scope. |
| Are open issues/PRs empty or intentionally deferred? | Yes, after this audit merges. | At audit start, `tcs-cosheaf-workspace-template` and `tcs-kb-public` had no open issues or PRs. `tcs-cosheaf` had only this audit issue, #257, and no open PRs. |

## Scope That May Be Claimed In v0.2.2

The release candidate may claim these implemented capabilities if the final
release-candidate verification passes:

- optional OpenAI-compatible HTTP transport, default-off;
- explicit `provider real-run` CLI path with consent, network, config, and
  context-preview checks;
- provider context-send policy matrix with public/private consent boundaries;
- provider log leak scanner and redacted provider run records;
- WorkerBundle v2 failure/counterexample preservation;
- role contracts for uncertainty, failures, verifier requests, candidates, and
  no-claim-invention librarian summaries;
- provider malformed-output recovery and provider workflow evals;
- failure/counterexample workflow evals;
- workspace real-provider setup docs and public-preview smoke;
- public KB source-note/backlog refresh and one draft-only foundation
  tightening;
- optional read-only MCP adapter review.

## Scope That Must Not Be Claimed

The release candidate must not claim:

- production hosted multi-agent platform readiness;
- default real provider calls;
- CI live provider coverage;
- hosted worker CLI commands;
- provider MCP tools;
- controlled-write MCP;
- web UI, SaaS behavior, or multi-user permissions;
- automatic theorem proving;
- full Lean/mathlib/CSLib integration;
- automatic informal/formal semantic alignment;
- automatic accepted promotion;
- AI/provider/MCP output as human review, source evidence, verifier evidence,
  gate success, or accepted knowledge.
