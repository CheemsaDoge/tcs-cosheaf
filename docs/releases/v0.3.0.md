# v0.3.0 - Checked Evidence + Research Run Loop

Status: release candidate. Package metadata and `cosheaf.__version__` record
`0.3.0` in the release-candidate PR. The public `v0.3.0` tag, GitHub release,
release smoke from `@v0.3.0`, and downstream workspace/public-KB pin updates
are intentionally deferred to the publication closeout after this PR merges.

This release packages the checked-evidence and research-run loop work that
landed after the published `v0.2.4` Artifact Failure Memory + Attempt
Traceability release. It turns candidate counterexamples and external-operator
work into reviewable, typed, repository-local evidence surfaces while
preserving the existing validation, gate, human review, public/private,
verifier, provider, MCP, and promotion boundaries.

This release is not production-ready software. It does not add automatic
theorem proving, full Lean/mathlib/CSLib proof checking of informal
statements, automatic autoformalization, automatic informal/formal semantic
alignment, automatic accepted promotion, AI-as-human-review, a web UI, SaaS
behavior, multi-user permissions, default-on provider calls, or provider/MCP
authority expansion.

## Highlights

- Checked counterexample evidence:
  `CheckedCounterexampleEvidenceRecord` and
  `schemas/counterexample_evidence.schema.json` provide a durable evidence
  record that is separate from WorkerBundle counterexample candidates, failure
  memory, verifier requests, verifier evidence, human review, accepted
  refutation, accepted status, and promotion authority.
- Controlled checked-evidence CLI:
  `cosheaf counterexample evidence validate/show/stage` validates or stages
  evidence only under `reviews/evidence/checked-counterexamples/`, with
  dry-run support and accepted-path, path-traversal, duplicate, readonly-root,
  and authority-spoofing protections.
- Context and readiness surfacing:
  context packs and promotion-readiness reports show checked counterexample
  evidence as review context only. Public-only context excludes private
  checked-evidence text and private target artifact IDs.
- Research-run provenance:
  `ResearchRunRecord` and `schemas/research_run.schema.json` record
  repository-local external-operator runs, including command records,
  artifacts read or touched, controlled outputs, evidence references, reports,
  limitations, and operator notes.
- Research-run CLI:
  `cosheaf run start/append-command/append-artifact/append-output/finalize/show/evidence-report/export-review/replay-plan`
  supports a full run lifecycle. Runtime records stay under `.cosheaf/runs/`;
  `export-review` is the controlled path into `reviews/runs/`.
- External operator workflow:
  docs and templates describe the CLI/Git loop for Codex-style agents without
  embedding GPT, Claude, hosted providers, or MCP as the default runtime.
- Downstream demo and policy:
  workspace-template includes a research-run demo, and public KB policy
  explains checked evidence as review evidence only.
- Integration and eval coverage:
  default evals and the ecosystem matrix cover candidate-vs-checked
  separation, checked-evidence support requirements, skipped-not-pass behavior,
  research-run completeness, evidence-report boundaries, replay-plan
  read-only behavior, private-leak prevention, authority escalation, workspace
  demo coverage, and public KB checked-evidence policy smoke.

## Boundaries

- Checked counterexample evidence is evidence for review. It is not human
  review, accepted refutation, accepted status, verifier pass, gate pass,
  source review, or promotion authorization.
- A candidate counterexample remains a candidate until a separate checker,
  review record, or evidence path supports a checked-evidence record.
- Research-run records are provenance. They are not proof, human review,
  verifier pass, gate pass, accepted status, or promotion authorization.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, verifier evidence where applicable, source metadata for
  accepted public artifacts, and explicit promotion.
- Skipped verifier, provider, SAT, SMT, Lean, lake, optional-tool, network, or
  operator results are not passes.
- A successful Lean `#check` means only import and symbol resolution. It does
  not prove informal/formal semantic alignment.
- Provider real-run behavior remains explicit and default-off.
- MCP remains optional adapter work and is not the primary release path.
- Public KB accepted content still requires source metadata and human review.

## Readiness Audit Summary

The readiness audit found no release blocker for entering this release
candidate. It confirmed:

- checked evidence is implemented and clearly separated from candidate
  counterexamples;
- checked evidence remains review evidence only and cannot create human
  review, accepted refutation, accepted status, or promotion authority;
- research-run records are reproducible, repository-local, redacted, and safe
  to review;
- Codex-style agents can operate the loop through CLI/Git without an embedded
  default model runtime;
- workspace-template and public KB have the needed demo and policy surfaces;
- security and deterministic eval suites are in place for authority
  escalation, private leakage, skipped-not-pass, and secret redaction;
- the default ecosystem matrix does not require hosted providers, API keys,
  MCP, SAT, SMT, Lean, lake, or network access; and
- open issues and PRs were absent before the release-candidate issue was
  created.

Deferred non-blockers:

- The public `v0.3.0` tag and GitHub release are publication-closeout work.
- Release smoke from `@v0.3.0` must run after the tag exists.
- Workspace-template and public KB active pins stay on `@v0.2.4` until
  `v0.3.0` release smoke passes.
- Optional SAT, SMT, Lean, lake, hosted-provider, MCP, and network paths remain
  optional; unavailable tools or network rows must stay `skipped`, not `pass`.

## Known Limitations

- No hosted provider call is default-on.
- No real provider call runs in CI or default tests.
- SAT, SMT, plain Lean, and external Lean reference checking remain minimal
  optional invocation paths.
- External Lean library reference checking does not fetch CSLib/mathlib or
  prove semantic alignment.
- Checked evidence is useful review context, not accepted mathematical truth.
- Research-run records are useful provenance, not proof or review authority.
- External public KB integration remains through local workspace roots, not a
  hosted registry service.
