# v0.2.1 - CLI Agent Access + Hosted Provider Gateway RC

`v0.2.1` prepares the post-`v0.2.0` local-MVP framework as a release
candidate for CLI-first agent access and the hosted provider gateway. It is a
pin-able framework candidate for coding-agent workflows, deterministic fake
provider runs, and reviewable provider-worker orchestration.

This release candidate is not a production-readiness claim. It does not add a
web UI, SaaS runtime, multi-user permissions, automatic theorem proving,
automatic Lean autoformalization, automatic accepted promotion, or automatic
informal/formal semantic alignment.

## Highlights

- CLI-first agent access:
  core read/check commands provide deterministic `--json` output for coding
  agents, humans, and CI.
- Controlled write CLI:
  draft artifact, staged source-note, worker-bundle submission, and draft
  review-request commands reject accepted writes, readonly KB roots, accepted
  artifact status, and human-review spoofing.
- Service-layer boundary:
  CLI, provider workers, orchestrator code, tests, and optional future MCP
  surfaces share typed service entry points instead of duplicating policy
  logic.
- Hosted provider gateway:
  deterministic fake provider and OpenAI-compatible mocked transport paths are
  implemented with secret redaction, timeout/retry/cancellation metadata,
  run-record logs, and no real network requirement in CI.
- Role-specific hosted workers:
  reasoner, verifier, counterexampleer, explorer, formalizer, and
  librarian-summarizer roles validate provider output as WorkerBundle v2 or
  typed review-only sub-results.
- Internal orchestrator hosted-worker dispatch:
  `cosheaf orchestrator run --issue <issue-id> --provider fake --json` runs a
  deterministic hosted-worker path without hosted network access, and the
  OpenAI-compatible boundary requires explicit configuration and consent.
- Security and evaluation coverage:
  agent-access security regressions and the Python-level agent workflow eval
  suite cover accepted-write rejection, malformed bundle rejection, context
  privacy, provider redaction, gate regressions, and optional read-only MCP
  smoke behavior.
- Operator guidance:
  the CLI-first operator contract and optional Skill package document how
  agents should use Cosheaf without widening authority.

## Boundaries

- MCP is optional adapter work. It is not required for `v0.2.1` workflows.
- Built-in real hosted HTTP transport is still not enabled by default and is
  not required for tests or CI.
- Real provider calls, when configured later, require explicit policy scope,
  preview, credentials, consent, and redacted audit records.
- Provider output is untrusted. It can become WorkerBundle output, draft
  proposal, draft artifact input, or review context only.
- Accepted knowledge still requires validation, gates, human review where
  policy requires it, and explicit promotion.
- AI review is not human review.
- Validation, gate, retrieval, context, provider, MCP, SAT, SMT, or Lean output
  is not accepted status by itself.
- Skipped verifier or provider results are not passes.
- Formal links remain metadata unless a checker actually runs and records a
  result.
- A successful external Lean `#check` means only that the import and symbol
  resolved; it does not prove informal/formal semantic alignment.

## Downstream Guidance

After the `v0.2.1` tag is published, downstream workspaces that need the
CLI-agent and hosted-provider gateway surfaces should pin to:

```text
git+https://github.com/CheemsaDoge/tcs-cosheaf.git@v0.2.1
```

Use `tcs-cosheaf-workspace-template` as the user-facing entry point. Mount or
checkout `tcs-kb-public` as readonly public knowledge and keep private work in
the writable private KB overlay.

Before depending on this release candidate for a downstream showcase, run the
framework command ladder, the workspace-template demo/showcase commands, and
the public KB validation/gate checks against the candidate source or tag.
